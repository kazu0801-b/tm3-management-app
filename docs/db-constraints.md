# DB-07 MVPで必要なDB制約ルール

## 目的

MVPで必要なDB制約ルールを整理し、API担当者・フロント担当者と認識合わせできる状態にする。

DB制約とは、データの不整合を防ぐためのルールです。  
例えば、必須項目を空で登録できないようにしたり、存在しないプロジェクトにタスクを紐づけないようにするために使用します。

## 前提

- users / roles は独自テーブルとして作成しない
- ユーザー管理・ロール管理は Directus 標準機能を使用する
- DB担当が整理する対象は主に以下のテーブル
  - projects
  - tasks
  - statuses
  - project_members
- MVPでは、複雑な制約よりも最低限データ不整合を防ぐルールを優先する

---

## 1. メールアドレスの重複禁止

### 対象

- Directus標準ユーザー

### ルール

| 項目 | ルール |
|---|---|
| メールアドレス | 重複不可 |
| メールアドレス | 必須 |

### 理由

同じメールアドレスのユーザーが複数存在すると、ログイン時にどのユーザーか判断できなくなるため。

### API・フロント向けメモ

- ユーザー作成時、同じメールアドレスがすでに存在する場合はエラーにする
- フロント側では「このメールアドレスはすでに使用されています」のようなエラー表示を想定する

---

## 2. projects テーブルの制約

### 対象テーブル

- projects

### 必須項目

| カラム | 必須 | 重複不可 | 補足 |
|---|---|---|---|
| id | 必須 | 重複不可 | プロジェクトを識別するID |
| name | 必須 | なし | プロジェクト名 |
| description | 任意 | なし | プロジェクト説明 |
| created_by | 必須 | なし | 作成者。Directus標準ユーザーを参照 |
| created_at | 必須 | なし | 作成日時 |
| updated_at | 任意 | なし | 更新日時 |

### 外部キー

| カラム | 参照先 | ルール |
|---|---|---|
| created_by | Directus標準ユーザー | 存在するユーザーのみ設定可能 |

### API・フロント向けメモ

- プロジェクト名は必須入力にする
- 作成者はログイン中のユーザーを自動設定する想定
- MVPではプロジェクト名の重複は許可してもよい

---

## 3. statuses テーブルの制約

### 対象テーブル

- statuses

### 必須項目

| カラム | 必須 | 重複不可 | 補足 |
|---|---|---|---|
| id | 必須 | 重複不可 | ステータスID |
| name | 必須 | 重複不可 | ステータス名 |
| sort_order | 必須 | 重複不可 | 表示順 |

### 初期データ

| id | name | sort_order |
|---|---|---|
| 1 | 未対応 | 1 |
| 2 | 対応中 | 2 |
| 3 | 確認中 | 3 |
| 4 | 対応済み | 4 |

### API・フロント向けメモ

- カンバン表示では `sort_order` の順番で列を表示する
- タスク作成時の初期ステータスは「未対応」とする
- MVPではステータス名は上記4種類で固定する

---

## 4. project_members テーブルの制約

### 対象テーブル

- project_members

### 必須項目

| カラム | 必須 | 重複不可 | 補足 |
|---|---|---|---|
| id | 必須 | 重複不可 | プロジェクト参加者ID |
| project_id | 必須 | なし | 参加するプロジェクト |
| user_id | 必須 | なし | 参加するユーザー |
| role | 必須 | なし | プロジェクト内の役割 |

### 外部キー

| カラム | 参照先 | ルール |
|---|---|---|
| project_id | projects.id | 存在するプロジェクトのみ設定可能 |
| user_id | Directus標準ユーザー | 存在するユーザーのみ設定可能 |

### 重複禁止ルール

| 組み合わせ | ルール |
|---|---|
| project_id + user_id | 同じプロジェクトに同じユーザーを重複登録しない |

### API・フロント向けメモ

- 同じユーザーを同じプロジェクトに2回登録しない
- タスク担当者の候補は、基本的に project_members から取得する
- MVPでは role は「管理者」「メンバー」程度でよい

---

## 5. tasks テーブルの制約

### 対象テーブル

- tasks

### 必須項目

| カラム | 必須 | 重複不可 | 補足 |
|---|---|---|---|
| id | 必須 | 重複不可 | タスクID |
| project_id | 必須 | なし | 紐づくプロジェクト |
| title | 必須 | なし | タスク名 |
| description | 任意 | なし | タスク詳細 |
| status_id | 必須 | なし | ステータス |
| assignee_id | 任意 | なし | 担当者 |
| start_date | 任意 | なし | 開始日 |
| due_date | 任意 | なし | 期限日 |
| created_by | 必須 | なし | 作成者 |
| created_at | 必須 | なし | 作成日時 |
| updated_at | 任意 | なし | 更新日時 |

### 外部キー

| カラム | 参照先 | ルール |
|---|---|---|
| project_id | projects.id | 存在するプロジェクトのみ設定可能 |
| status_id | statuses.id | 存在するステータスのみ設定可能 |
| assignee_id | Directus標準ユーザー | 存在するユーザーのみ設定可能 |
| created_by | Directus標準ユーザー | 存在するユーザーのみ設定可能 |

### API・フロント向けメモ

- タスクは必ず1つのプロジェクトに紐づける
- タスクは必ず1つのステータスに紐づける
- 担当者は未定の場合もあるため、MVPでは任意でもよい
- タスク作成時の初期ステータスは「未対応」とする

---

## 6. 開始日と期限日のルール

### 対象

- tasks.start_date
- tasks.due_date

### ルール

| 項目 | ルール |
|---|---|
| start_date | 任意 |
| due_date | 任意 |
| start_date と due_date の両方がある場合 | start_date は due_date 以前の日付にする |
| due_date のみある場合 | カレンダー表示に使用可能 |
| start_date と due_date の両方がある場合 | 簡易ガント表示に使用可能 |

### OK例

| start_date | due_date | 判定 |
|---|---|---|
| 2026-05-24 | 2026-05-26 | OK |
| 空 | 2026-05-26 | OK |
| 2026-05-24 | 空 | OK |

### NG例

| start_date | due_date | 判定 |
|---|---|---|
| 2026-05-27 | 2026-05-24 | NG |

### API・フロント向けメモ

- フロント側では、開始日が期限日より後にならないように入力制御する
- API側でも、開始日が期限日より後の場合はエラーにする
- カレンダー表示では due_date を使用する
- 簡易ガント表示では start_date と due_date を使用する

---

## 7. タスクとプロジェクトの紐づき

### ルール

| 内容 | ルール |
|---|---|
| タスクはプロジェクトに属する | tasks.project_id は必須 |
| 存在しないプロジェクトへの登録 | 不可 |
| プロジェクト削除時の扱い | MVPでは要確認 |

### API・フロント向けメモ

- タスク作成時は project_id を必ず指定する
- プロジェクト詳細画面やタスク一覧画面では、project_id を条件にタスクを取得する
- プロジェクト削除をMVPで扱う場合、紐づくタスクをどうするかは別途確認する

---

## 8. タスクとステータスの紐づき

### ルール

| 内容 | ルール |
|---|---|
| タスクはステータスを持つ | tasks.status_id は必須 |
| 存在しないステータスへの登録 | 不可 |
| 初期ステータス | 未対応 |

### API・フロント向けメモ

- カンバン表示では status_id ごとにタスクを分類する
- ステータス変更時は tasks.status_id を更新する
- ステータス選択肢は statuses から取得する

---

## 9. タスクと担当者の紐づき

### ルール

| 内容 | ルール |
|---|---|
| 担当者 | MVPでは任意 |
| 担当者に設定できる人 | 基本的にプロジェクト参加者 |
| 存在しないユーザーへの登録 | 不可 |

### API・フロント向けメモ

- 担当者候補は project_members から取得する
- 担当者未設定のタスクも許可するかはチームで確認する
- MVPでは、1タスクにつき担当者1名を基本とする

---

## 10. MVPでの制約まとめ

| 対象 | ルール |
|---|---|
| メールアドレス | 重複不可 |
| プロジェクト名 | 必須 |
| ステータス名 | 必須・重複不可 |
| ステータス表示順 | 必須・重複不可 |
| タスク名 | 必須 |
| タスクとプロジェクト | 必ず紐づける |
| タスクとステータス | 必ず紐づける |
| タスクと担当者 | MVPでは任意、設定する場合は存在するユーザー |
| 開始日・期限日 | 開始日は期限日以前 |
| 同一プロジェクト内の同一メンバー | 重複登録しない |

---

## 11. 要確認事項

| No | 確認事項 | 補足 |
|---|---|---|
| 1 | 担当者未設定のタスクを許可するか | MVPでは許可でもよい |
| 2 | プロジェクト削除時にタスクも削除するか | MVPでは削除機能自体を後回しでもよい |
| 3 | プロジェクト名の重複を許可するか | MVPでは許可でもよい |
| 4 | role を project_members に文字列で持つか | Directus標準ロールとの兼ね合いを確認 |
| 5 | 開始日・期限日の必須条件 | ガントを必須にするなら両方必要なタスクを用意する |

---

## 12. 完了条件

- MVPで必要な最低限のDB制約ルールが整理されている
- 必須項目が整理されている
- 重複不可の項目が整理されている
- 外部キーの考え方が整理されている
- 開始日と期限日のルールが整理されている
- API担当者、フロント担当者と認識合わせできる状態になっている
