# DB設計README

## 目的

TM3管理ツールのDB設計内容を、後から見返せるようにまとめる。

DB担当・API担当・フロント担当が、テーブル構成や設計意図を確認できるようにする。

---

## 前提

TM3管理ツールでは、Backlogのようなプロジェクト・タスク管理アプリをMVPとして作成する。

技術方針は以下とする。

- フロントエンド：Next.js / TypeScript / Tailwind CSS
- バックエンド：Directus
- DB：MySQL
- 認証・ユーザー管理：Directus標準機能を利用

---

## Directus利用方針

API担当との確認結果により、以下の方針とする。

- `users` は自作テーブルとして作成しない
- `roles` は自作テーブルとして作成しない
- ユーザー管理・ロール管理は Directus 標準機能を利用する

そのため、DB担当が主に設計対象とするテーブルは以下とする。

- `projects`
- `project_members`
- `statuses`
- `tasks`

---

## テーブル一覧

| テーブル名 | 役割 |
|---|---|
| projects | プロジェクト情報を管理する |
| project_members | プロジェクトに参加しているユーザーを管理する |
| statuses | タスクの進捗ステータスを管理する |
| tasks | プロジェクトに紐づくタスク情報を管理する |

---

## 各テーブルの役割

### projects

プロジェクト情報を管理するテーブル。

主な用途：

- プロジェクト一覧表示
- プロジェクト作成
- タスクをどのプロジェクトに紐づけるかの管理

主な項目：

- プロジェクトID
- プロジェクト名
- 説明
- 作成者
- 作成日時
- 更新日時

---

### project_members

プロジェクトに参加しているユーザーを管理するテーブル。

主な用途：

- プロジェクト参加者の管理
- タスク担当者候補の取得
- プロジェクトごとのメンバー確認

主な項目：

- プロジェクト参加者ID
- プロジェクトID
- ユーザーID
- プロジェクト内の役割

補足：

ユーザーIDは Directus 標準ユーザーのIDを参照する想定。

---

### statuses

タスクの進捗状態を管理するテーブル。

主な用途：

- タスク一覧でのステータス表示
- カンバン列の表示
- タスク作成・編集時のステータス選択

初期ステータス：

| id | name | sort_order |
|---|---|---|
| 1 | 未対応 | 1 |
| 2 | 対応中 | 2 |
| 3 | 確認中 | 3 |
| 4 | 対応済み | 4 |

---

### tasks

プロジェクトに紐づくタスクを管理するテーブル。

主な用途：

- タスク一覧表示
- タスク作成・編集・削除
- カレンダー表示
- カンバン表示
- 簡易ガント表示

主な項目：

- タスクID
- プロジェクトID
- タスク名
- 説明
- ステータスID
- 担当者ID
- 開始日
- 期限日
- 作成者
- 作成日時
- 更新日時

補足：

担当者ID、作成者IDは Directus 標準ユーザーのIDを参照する想定。

## 初期データの説明

開発確認用の初期データは、以下の資料にまとめている。

- [INSERT文](./seed-data.sql)

確認できる内容：

- タスク一覧表示
- カレンダー表示
- カンバン表示
- 簡易ガント表示
- 担当者表示
- プロジェクト参加者表示

補足：

`users` / `roles` は自作テーブルではなく Directus 標準機能を利用するため、INSERT文には含めない。

`projects.created_by`、`project_members.user_id`、`tasks.assignee_id`、`tasks.created_by` には、Directus標準ユーザーIDを入れる想定。

---

## 要確認事項

| No | 確認事項 | 補足 |
|---|---|---|
| 1 | Directus標準ユーザーIDの型 | SQLでは仮で `CHAR(36)` としている |
| 2 | Directus標準ユーザーとのリレーション設定方法 | Directus上の設定方法をAPI担当・バックエンド担当と確認 |
| 3 | 担当者未設定タスクを許可するか | MVPでは許可する方針 |
| 4 | プロジェクト削除時に紐づくタスクをどうするか | MVPでは削除機能を後回しにする可能性あり |
| 5 | 本番投入時の初期データ | 開発確認用データとは分けて考える |

## 今回やらないこと

MVPでは、以下は対象外とする。

| 対象外 | 理由 |
|---|---|
| コメント機能 | 実装コストが上がるため |
| 通知機能 | 実装コストが上がるため |
| ファイル添付 | DB・ストレージ設計が重くなるため |
| リアルタイム更新 | 初学者チームには負担が大きいため |
| 複数チーム管理 | 権限・データ設計が複雑になるため |
| ガントの依存関係 | 簡易ガントに留めるため |
| 高機能ドラッグ&ドロップ | MVPではクリック操作による更新を優先するため |

## DB設計の完了条件

DB設計として、以下が確認できる状態を目指す。

- テーブル一覧が確認できる
- 各テーブルの役割が確認できる
- ER図を確認できる
- テーブル定義書を確認できる
- 初期データの内容を確認できる
- API担当・フロント担当が必要なDB情報を確認できる
- 後から見ても設計意図が分かる
